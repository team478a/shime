# SHIME クライアント向け利用者デモガイド

作成日: 2026-07-17  
対象: SHIME staging（合成データ限定）  
所要時間: 10〜15分

## 1. デモで伝えること

SHIMEは、申込みから本人連携、イベント準備、当日受付、席案内、希望入力、結果確認までをLINE中心でつなぐ運営OSである。

デモでは、利用者が何度も氏名やイベント情報を入力し直さず、スマートフォンだけで次の行動へ進めることを見せる。

## 3. 推奨デモシナリオ

検証イベント `RH-B` と未連携の合成参加者を使用する。本番参加者や本番LINE公式アカウントのデータは使用しない。

### 3.1 LINE本人連携

1. 運営が管理画面から対象者の本人連携リンクを発行する。
2. 利用者がリンクをLINEで開く。
3. 申込時の電話番号下4桁または生年月日で本人確認する。
4. 「本人連携が完了しました」を確認する。

説明ポイント:

- URLは72時間で失効し、再発行すると旧URLは使えなくなる。
- 1イベント内で、同じLINEアカウントを複数参加者へ重複連携できない。
- 生のLINE IDや連携トークンを画面・ログへ表示しない。

### 3.2 感情カードとDream登録

1. 「Dream登録へ進む」を押す。
2. 今の気持ちに近い感情カードを選ぶ。
3. 第一印象、関連する領域、その奥にある願いを入力する。
4. 表示された候補を選択または編集する。
5. 公開範囲を選び、Dreamを確定する。

説明ポイント:

- AI未接続でも固定候補へ自動フォールバックし、利用者体験を停止させない。
- Dreamの公開範囲とプロジェクト参加は本人が選択する。
- 非公開Dreamを他の参加者へ表示しない。

### 3.3 席案内の5問

1. 5問に回答する。
2. 回答しづらい質問では「答えたくない」を選べることを見せる。
3. 必要に応じて途中保存する。
4. 5問を提出する。

説明ポイント:

- 回答は席配置の計算にのみ使用し、他参加者へ公開しない。
- 通信に失敗しても、保存済み回答を再入力させない設計を採用している。

### 3.4 SHIME PASSと受付

1. SHIME PASSを発行する。
2. 参加者番号と準備状況を確認する。
3. 受付QRを表示する。
4. スタッフ側でQR受付、または参加者番号の短い前方一致検索による手動受付を行う。
5. 利用者側で受付済み状態を確認する。

説明ポイント:

- QRには氏名、参加者番号、Dream No.を直接含めない。
- QRが表示できない場合も、参加者番号による手動受付へ切り替えられる。
- スタッフ画面はスマートフォン優先で、短い検索入力と候補選択を基本にしている。

### 3.5 席・希望・結果（進捗説明）

席公開後は、参加者が確定席を確認し、会話済みの相手だけを希望候補として選択する。希望は0名でも提出でき、一方希望、順位、任意メモを相手へ表示しない。

結果はシステムが自動公開せず、運営責任者が双方希望候補と競合を確認して確定した後にだけ参加者へ表示する。

この区間は実装済みだが、複数LINEアカウントを使う一連のstaging実機リハーサルを完了してから本番可否を判定する。

## 5. クライアントへ強調する設計

- イベント名、日時、会場、フォーム、席、通知などを管理画面で設定できる。
- イベント固有値をコードへ固定せず、複数イベント・複数サービスへ横展開できる。
- 参加者・スタッフともスマートフォンを優先し、入力回数を減らしている。
- 受付、席、希望判定は生成AIへ依存しない。
- 席と結果は責任者確認前に参加者へ公開しない。
- テナントとイベントの境界を全処理で維持する。
