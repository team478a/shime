# SHIME 婚活イベント 開発実装指示書 v1.0

- 基準仕様：`DEVELOPMENT_SPEC_V2.md`
- 本番目標：2026年8月8日
- 目的：開発担当者がIssueへ分割し、順番どおり実装できる状態にする

## 1. 開発の基本ルール

1. 参加者画面はスマートフォン・LINE内ブラウザを最優先する。
2. イベント固有値をコードへ直書きしない。
3. 夢登録方式と希望入力方式は `events` の設定で切り替える。
4. 席配置と成立候補の計算は決定論的なルールで実装する。
5. AIの失敗で受付・席・結果処理を止めない。
6. 本番データを開発環境に入れない。
7. P0機能完成後までP1・P2を追加しない。
8. API、DB、画面の変更を同じPull Requestへ含める場合は、移行・ロールバック手順を記載する。

## 2. 推奨リポジトリ構成

```text
shime-concierge/
├─ apps/
│  ├─ web/                 # LIFF・申込フォーム・管理画面
│  └─ worker/              # 通知・非同期処理（必要時）
├─ packages/
│  ├─ db/                  # schema、migration、seed
│  ├─ core/                # tenant、event、権限、状態遷移
│  ├─ line/                # LIFFトークン検証、Messaging API
│  ├─ application/         # 申込み・CSV
│  ├─ dream/               # 感情カード、夢、Dream No.
│  ├─ passport/            # QR・受付
│  ├─ seating/             # 採点・配置
│  ├─ matching/            # 希望・候補・結果
│  └─ ui/                  # 共通UI
├─ docs/
├─ tests/
└─ scripts/
```

単一Next.js構成で開始する場合も、上記のモジュール境界に相当するディレクトリ分離を保つ。

## 3. マイルストーン

| マイルストーン | 内容 | 完了基準 |
|---|---|---|
| M1 基盤 | DB、認証、権限、イベント設定 | 管理者がイベントを作成できる |
| M2 申込み | SHIMEフォーム、CSV、参加者 | 2経路が同一参加者管理へ入る |
| M3 LIFF・Dream | LINE紐づけ、カード、夢 | 必須・任意の両導線を通せる |
| M4 Passport | Passport、QR、受付 | QR・手動受付が動く |
| M5 席案内 | 5問、採点、配置、公開 | 欠席反映後に席を確定できる |
| M6 希望・結果 | 3方式、候補、確定、通知 | 3方式の受入テストが通る |
| M7 本番準備 | 総合試験、リハーサル | 代替運用を含む通し試験完了 |

### 横断タスク：現地スマートフォン運用

- 受付、取消、本人連携再開、席操作、結果確認をスマートフォン単体で完結させる
- 参加者選択は番号の前方一致、氏名の部分一致、候補選択を共通仕様にする
- 理由入力は代表的な選択肢と任意補足入力を用意する
- 一覧表はモバイルでカードまたは重要列優先表示にする
- 破壊的・公開・通知操作は、入力を増やす代わりに対象数・対象イベント・結果を確認画面で明示する
- 実機テストはスマートフォン幅390 pxを基準に、主操作、キーボード表示、横はみ出し、タップミスを確認する

| 優先度 | 既存操作の改善対象 | スマートフォン向け改善 |
|---|---|---|
| P0 | 手動受付 | 番号前方一致・氏名部分一致の候補選択（実装済み） |
| P0 | 受付取消 | 取消理由プリセット＋任意補足、対象参加者の再入力なし（実装済み） |
| P0 | 本人連携・参加者一覧 | 短い検索、未連携・連携済みフィルター、モバイルカード表示（実装済み） |
| P0 | 管理トップ・ナビゲーション | イベントごとの当日受付・席配置直接ボタン、スマホ用開閉メニュー、資料リンク折りたたみ（実装済み） |
| P1 | 席配置調整 | 参加者検索、配置状態の絞り込み、使用中席の選択防止、モバイルカード、テーブル別座席図（実装済み・座席図は実機確認待ち） |
| P1 | 結果確定・通知 | 対象数と状態の要約、候補フィルター、プレビュー後の一操作実行 |

## 5. API実装共通事項

- 入出力をスキーマ検証する
- 変更処理はトランザクションを使う
- 申込み、受付、結果確定、通知に冪等性を持たせる
- エラー応答へ `request_id` を付ける
- PIIをログへ出さない
- tenant_id／event_idの境界テストを全管理APIに用意する

## 6. テストデータ

seedで以下を作る。

- テナント1件
- 8月8日イベント1件
- 受付、運営、責任者、システム管理者各1名
- 参加者50名
- 新規30名、既存Dream No.20名
- 5問回答済み40名、未回答10名
- 受付済み、未受付、欠席を含む
- 接触回避2組
- 感情カード最低8枚
- 希望入力3方式それぞれのテストイベント

## 7. Definition of Done

各Issueは次を満たして完了とする。

- 受入条件を満たす
- unit／integration testを追加
- 権限・テナント境界を確認
- スマートフォン表示を確認
- エラー・空状態・読込中状態がある
- 監査対象操作はログへ残る
- 設定値をコードへ直書きしていない
- 仕様書またはAPI資料を更新
- レビュー承認済み

## 8. 8月8日までの削減ルール

遅延時は次の順で削減し、P0導線を守る。

1. 席配置のドラッグ操作をフォーム式入替へ変更
2. 通知予約を手動送信へ変更
3. 申込フォーム項目の自由カスタマイズを固定項目へ縮小
4. 重複候補の高度な統合UIを一覧警告へ縮小
5. AI夢候補を固定テンプレートへ切替

削減してはいけないもの：

- LINE本人紐づけの安全性
- 同意記録
- QR・手動受付
- 席の手動確定
- 希望内容の非公開
- 結果の運営確定
- CSVバックアップ

## 9. 本番リハーサル手順

1. 本番相当CSVを検証環境へ匿名化して取込
2. 新規・既存・Dream任意スキップを各2名実施
3. LINE連携、Passport、5問を完了
4. 当日受付端末でQR・手動受付
5. 欠席者を発生させて席再計算
6. 席を手動変更して公開
7. 希望入力と双方希望候補を確認
8. 結果通知をテスト用LINEだけに送信
9. AIを停止して同じ導線を再実施
10. CSV・紙の受付表・席表で代替運用確認
