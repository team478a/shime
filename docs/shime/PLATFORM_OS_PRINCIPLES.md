# SHIME運営OS 設計原則

更新日: 2026-07-17

## 1. 方針

SHIMEを単一の婚活イベント管理システムとして閉じず、複数の事業、イベント、顧客組織、外部サービスを組み合わせて利用できる運営OSとして設計する。

ここでいうOSは端末用のオペレーティングシステムではなく、イベント・コミュニティ・顧客接点の運営に共通する機能を提供する業務プラットフォームを意味する。

## 2. スコープ階層

1. **Platform**: システム全体の共通機能と運用基盤
2. **Tenant**: 運営会社・顧客組織ごとのデータと権限境界
3. **Service / Module**: 婚活、Dream支援などの事業・機能単位
4. **Reusable Resource**: 会場、フォーム、通知文、カードセットなどの再利用可能な版付きテンプレート
5. **Event / Campaign**: 実際に実施するイベントまたは施策
6. **User / Participant**: 同意と権限に基づき各サービスへ関与する利用者

すべてのデータ取得と更新で必要なスコープを明示する。単にIDが一致することだけを認可条件にしない。

## 3. 共通プラットフォーム機能

- 認証・本人確認
- ロールと権限
- 監査ログ
- 動的フォームと同意文書
- 通知テンプレートと送信基盤
- 定期ジョブと運用監視
- ファイル・CSV・画像管理
- LINE、AI、Storageなどの外部接続
- テンプレート、版、公開状態、イベントへのスナップショット

共通基盤へ婚活固有の参加区分、希望順位、成立条件などを直接埋め込まない。

## 4. サービスモジュール

各サービスモジュールは次を明示する。

- 有効化に必要な設定
- 使用する権限
- 提供する管理画面と参加者画面
- 読み書きするデータのスコープ
- 他モジュールへ公開する契約
- 無効化時のフォールバックと履歴保持方法

モジュール同士が連携する場合は、内部テーブルの直接参照を増やさず、型付きインターフェース、API、ドメインイベントまたは版付きスナップショットを使用する。

## 5. テンプレートとイベント実体

会場構成、テーブル・席、申込フォーム、通知文、規約、カードセットは再利用可能なテンプレート候補とする。

- テンプレートには版と利用状態を持たせる
- イベント作成時または設定確定時にスナップショットを作る
- テンプレート変更は新しいイベントへだけ明示的に反映する
- 進行中・完了済みイベントをテンプレート更新で暗黙に変更しない
- イベント側の個別調整を許可し、元テンプレートを逆に書き換えない
- 適用履歴には元版と保存成功時の最終スナップショットを保持し、単なる画面上の選択は利用実績に数えない
- アーカイブは新規利用を止める操作とし、版、適用履歴、既存イベントのスナップショットを削除しない

テーブル初期設定では、汎用 `resource_templates` の `event_operations / venue_layout` をテナントの再利用資源として管理し、イベントにはテーブル・席のコピーを保持する。保存成功時の元版・最終構成・ハッシュは `resource_template_applications` に記録する。

## 6. 複合利用

複数機能を組み合わせる場合も、利用者の同意、権限、公開範囲を機能ごとに判定する。あるサービスへの参加を、別サービスへの参加や公開同意として扱わない。

共通ユーザーを利用できる場合でも、参加情報、Dream、希望、感情回答などの目的外共有は行わない。

## 7. 段階的な移行

2026年8月8日のイベント運営を最優先とし、既存P0を全面的に作り直さない。新しい機能から次の順に境界を整える。

1. 共通設定と業務固有設定を分離する
2. 再利用資源を版付きテンプレート化する
3. イベントへスナップショットする
4. モジュール登録と権限を明示する
5. モジュール間連携を契約化する

この原則に反する短期実装が必要な場合は、仕様逸脱と解消予定を記録する。
