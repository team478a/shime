# SHIME Codex開発開始指示書 v1.0

## 1. この文書の目的

本書は、Codexを使ってSHIME婚活イベントシステムを実装する際の最上位指示書である。

Codexは以下の資料を基準に実装する。

1. `DEVELOPMENT_SPEC_V2.md`
2. `IMPLEMENTATION_TASKS_V1.md`
3. `EVENT_CONFIG_20260808.yaml`
4. `APPLICATION_IMPORT_TEMPLATE.csv`
5. 承認済みクリックモック

仕様が競合する場合の優先順位は、ユーザーの最新指示、本書、開発仕様書v2.0、実装指示書、モックの順とする。

## 2. Codexへの基本指示

```text
あなたはSHIME婚活イベントシステムの実装担当です。

2026年8月8日のイベント本番を最優先とし、仕様書のP0機能を段階的に実装してください。

作業開始時に必ず以下を行ってください。

1. AGENTS.mdと指定仕様書をすべて読む
2. 現在のリポジトリ構成と既存実装を調査する
3. 未コミット変更を確認し、ユーザーの既存変更を保持する
4. 実装予定を短い計画として提示する
5. 1フェーズずつ実装、テスト、報告する

一度に全機能を実装しないでください。
イベント固有値をコードに直書きしないでください。
AIが停止してもイベント運営できる設計を維持してください。
個人情報をログへ出さないでください。
参加者の一方希望、順位、感情回答を他参加者へ表示しないでください。
席とマッチング結果は運営責任者が確定するまで公開しないでください。

各フェーズ終了時に、変更ファイル、実装内容、テスト結果、残課題、次フェーズを報告してください。
```

## 3. 推奨開発方式

### 3.1 リポジトリ

- SHIME専用GitHubリポジトリを1つ用意する
- `main`：本番
- `develop`：統合確認用。小規模開発では省略可能
- `feature/SHIME-xxx-short-name`：機能開発
- Pull Request単位でIssueを閉じる

既存の別製品リポジトリへ混在させない。共通化はSHIME Core内部で行う。

### 3.2 初期技術選定

- Next.js／React／TypeScript
- PostgreSQL
- ORMはPrismaまたはDrizzleのどちらか1つ
- Zod等による入出力検証
- LINE LIFF SDK
- LINE Messaging API
- Vitest等の単体・結合テスト
- Playwright等のE2Eテスト
- Vercelによる検証環境・本番環境

Codexはリポジトリに既存技術がある場合、合理的な理由なく置き換えない。

## 4. Codex用フェーズ分割

## Phase 0：リポジトリ調査・設計固定

### Codexへの指示

```text
リポジトリと仕様書を確認し、実装はまだ行わず、次を報告してください。

- 現在の技術構成
- 再利用できる既存コード
- 不足している基盤
- 仕様と既存実装の矛盾
- 推奨ディレクトリ構成
- DB migration方針
- 認証方式
- Phase 1で変更するファイル一覧

不明点のうち、実装結果を大きく変えるものだけ質問してください。
```

### 完了条件

- 技術選定が1つに決まる
- DB・認証・環境分離方針が決まる
- Phase 1の変更範囲が確定する

## Phase 1：Core・DB・イベント設定

### 対象Issue

- SHIME-001〜003
- SHIME-010〜012

### Codexへの指示

```text
Core基盤、PostgreSQL schema、migration、seed、スタッフ権限、イベント設定を実装してください。

このフェーズでは申込フォームやLIFF画面を実装しないでください。

最低限、次を作成してください。

- tenants
- users / user_identities
- staff_roles
- events
- event_form_fields
- event_tables / event_seats
- tenant_modules
- consents
- audit_logs
- 管理者ログインと管理API保護
- 8月8日テストイベントseed

夢登録方式と希望入力方式をイベント設定として保存してください。
状態遷移をサービス層で検証し、UIだけに依存させないでください。
```

### 必須テスト

- migrationを空DBへ適用
- seed実行
- ロール別API拒否
- イベント状態遷移
- tenant_id境界

## Phase 2：申込フォーム・CSV取込

### 対象Issue

- SHIME-020〜023

### Codexへの指示

```text
SHIME内申込フォームとCSV取込を実装してください。

両方の申込データは同じapplicationsテーブルへ保存してください。
CSVは事前検証と本取込を分け、検証だけではDBを変更しないでください。
再取込で受付・希望・結果を上書きしないでください。
重複候補を自動統合しないでください。
```

### 必須テスト

- 正常申込み
- 必須項目不足
- 二重送信
- CSV正常・警告・エラー
- CSV再取込差分
- 重複候補

## Phase 3：LINE・LIFF認証

### 対象Issue

- SHIME-030〜033

### Codexへの指示

```text
LIFF認証、申込みとの紐づけ、LINE Webhook、通知キューを実装してください。

クライアントでデコードしたプロフィール情報を本人確認済み情報としてサーバーへ送らないでください。
生のIDトークンをサーバーで検証してください。
Webhook署名を必ず検証してください。
link tokenはハッシュ保存し、利用後に失効してください。

LINE実環境がなくてもテストできるよう、LINE Provider Interfaceとテスト用Fake Providerを用意してください。
```

### 必須テスト

- 正しい・不正・期限切れIDトークン
- link tokenの利用・再利用・期限切れ
- 1申込み1LINE
- Webhook署名
- 通知二重送信防止

## Phase 4：感情カード・夢登録

### 対象Issue

- SHIME-040〜043

### Codexへの指示

```text
感情カード、新規3問、既存1問、夢登録、Dream No.、AI橋渡しを実装してください。

イベント設定に応じて以下を切り替えてください。

- required_private_allowed
- optional

任意の場合は夢を登録せずに進める導線を作ってください。
9999人夢応援プロジェクト同意をイベント必須同意と分離してください。
AI Gatewayと固定文Fallbackを実装してください。
```

### 必須テスト

- 夢必須・公開
- 夢必須・非公開
- 夢任意・登録
- 夢任意・スキップ
- AI成功・タイムアウト・失敗
- Dream No.重複防止

## Phase 6：席案内

### 対象Issue

- SHIME-060〜063

### Codexへの指示

```text
席案内5問、ペアスコア、初期席配置、運営手動調整、確定公開を実装してください。

席配置を生成AIへ決めさせないでください。

欠席、接触回避、参加区分、席定員をハード制約として必ず守ってください。
```

### 必須テスト

- 5軸個別採点
- 未回答時の重み正規化
- ハード制約
- 同点処理
- 手動ロック後の再計算
- manager以外の公開拒否

## Phase 7：希望入力・結果

### 対象Issue

- SHIME-070〜074

### Codexへの指示

```text
希望入力、双方希望候補、運営確定、結果通知を実装してください。

イベント設定に応じて次の3方式を切り替えてください。

- mutual_up_to_2
- first_choice_only
- ranked_up_to_3

一方希望、相手の順位、自由メモを参加者へ返さないでください。
システムは候補抽出までとし、managerが確定するまで結果を通知しないでください。
```

### 必須テスト

- 3方式の選択上限
- 順位重複禁止
- 双方希望候補
- 一方希望の非公開
- 複数成立設定
- 確定前通知拒否

## Phase 8：統合試験・本番準備

### Codexへの指示

```text
新機能を追加せず、統合試験と本番準備だけを行ってください。

- 新規・既存導線
- 夢必須・任意
- CSV・SHIMEフォーム
- QR・手動受付
- 欠席後の席再計算
- 希望3方式
- 結果通知
- AI停止
- LINE通知失敗
- CSVバックアップ

重大度をP0、P1、P2に分類し、P0が0件になるまで本番可能と判定しないでください。
```

## 5. Codexが勝手に変更してはいけない事項

- 夢公開をイベント参加必須にする
- 9999人夢応援プロジェクト参加を初期ONにする
- 一方希望を相手へ表示する
- AIでマッチング結果を自動確定する
- AIで席を直接決める
- QRへ氏名・連絡先・Dream No.を埋め込む
- イベント開始後に希望方式を自動変更する
- CSV再取込で受付・希望・結果を削除する
- 未承認の席・結果を参加者APIへ返す
- 本番の個人情報をログ・テスト・AIへ送信する
