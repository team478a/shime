# SHIME® 婚活AIコンシェルジュ 段階開発指示書レビュー

- レビュー日: 2026-07-23
- 対象文書: `SHIME_Codex_段階開発指示書.md`
- 対象リポジトリ: `team478a/shime`
- 対象ブランチ: `main`
- レビュー範囲: 仕様整合性、既存実装との衝突、セキュリティ・運用上の不足、実装前の確認事項
- この文書の位置づけ: 調査・判断記録。ソースコード、DB、画面の変更を承認するものではない

## 1. 結論

段階開発指示書の基本方針は、現在のSHIMEイベント運営OSおよびプラットフォーム化方針と概ね整合している。

特に、次の方針は維持する。

- 既存のイベント運営機能を置き換えず、追加モジュールとして実装する
- 既存の「席案内5問」と新規の「SHIME診断4問」を分離する
- AI障害で受付、席配置、希望入力、結果確定、通知を停止させない
- テナント、サービス／モジュール、イベントの境界をすべての処理で検査する
- テンプレートとイベント利用時点のスナップショットを分離する
- スマートフォンを参加者・現地スタッフ双方の主要端末として扱う
- 各Phaseを別PRとし、テストと承認後に次へ進む

ただし、現状のままPhase 1以降へ進むと既存テーブルとの衝突、診断回答の公開範囲、AIへのデータ送信、チャット運用、決済条件、本番イベント準備との優先順位で問題が生じる。まず「Concierge Phase 0」として差分調査を実施し、本書の確認事項を確定させる必要がある。

## 2. 既存方針と一致している点

| 項目 | 確認結果 |
| --- | --- |
| 追加開発 | 既存OSを維持し、機能をモジュールとして追加する方針と一致 |
| 5問と4問の分離 | 席配置の決定ロジックと本人向け診断を混在させないため必須 |
| AI Provider分離 | 既存のProvider／Fallback方針と一致 |
| AI非依存運営 | AI停止中もイベントの主要処理を継続する既存要件と一致 |
| テナント・イベント境界 | 現行のデータアクセス原則と一致 |
| Migration | スキーマ変更をMigrationで管理する既存ルールと一致 |
| スナップショット | テンプレート更新が進行中・過去イベントを変更しないOS原則と一致 |
| スマートフォン優先 | 現地運営でPCを前提としない既存要件と一致 |
| 決定ロジック | 希望順位やマッチングをAIに決めさせない方針と一致 |
| 段階的PR | 変更範囲とロールバック単位を小さくする方針と一致 |

## 3. 矛盾・衝突事項

### C-01: Phase番号が既存計画と重複する

既存の開発計画にもPhase 0〜8が存在する。新指示書のPhase 0〜6を同じ名称で管理すると、本番準備、PR、完了記録、課題管理でどちらのPhaseか判別できない。

**推奨対応**

- 新指示書のPhaseを `Concierge Phase 0`〜`Concierge Phase 6` と表記する
- 既存計画を `Event OS Phase 0`〜`Event OS Phase 8` と表記する
- PR名、ブランチ名、Migration名、完了記録にも区分を含める

### C-02: 推奨された `emotion_cards` が既存テーブルと衝突する

現行DBにはDream機能で使用中の次のテーブルがすでに存在する。

- `emotion_card_sets`
- `emotion_cards`
- `emotion_selections`

新指示書の `emotion_cards` は `base_emotion`、`intensity`、`composite_emotions`、`service_module` など別の意味と項目を持つ。既存テーブルの意味を変更すると、Dream登録、感情カードAPI、管理画面、履歴データへ影響する。

**推奨対応**

Concierge Phase 0で既存カードを再利用できる範囲を調査し、少なくとも初期実装では名前を分離する。

候補:

- `diagnosis_emotion_card_sets`
- `diagnosis_emotion_cards`
- `diagnosis_emotion_mappings`

将来共通化する場合も、既存データを意味変更せず、明示的な移行計画と互換レイヤーを用意する。

### C-03: 既存の感情入力とSHIME診断4問の境界が未定義

現行のDream導線には、感情カード選択と複数の内省入力が存在する。新しい診断4問と画面上・概念上で似ているため、参加者が同じ質問へ二度回答しているように感じる可能性がある。

**確認が必要な内容**

- Dream登録とSHIME診断を実施する順序
- 両方を必須にするか、イベントごとに有効／無効を選べるか
- 既存回答を新診断へ自動転用しないこと
- 表示名、説明文、進捗表示で別機能だと理解できること
- Love Passportにどの情報を表示するか

### C-04: 診断テンプレートのスコープが不足している

`diagnosis_templates` がプラットフォーム共通、テナント所有、サービス所有のどれかが明確でない。イベントだけでなくサービス／モジュール境界も必要である。

**推奨対応**

- テンプレート所有者と利用可能範囲を明示する
- `tenant_id`、`service_module` または同等のモジュール識別子を持たせる
- イベントではテンプレート最新版を直接参照せず、不変スナップショットを参照する
- 表示ラベル、質問、選択肢、カード、レポート設定を同じ版として固定する

### C-05: 診断回答の管理者公開範囲が既存プライバシー原則と衝突する

新指示書には「本人と権限を持つ管理者以外に表示されない」とある。一方、SHIMEでは私的なDream・感情・一方的な希望を不用意に運営者や他参加者へ公開しないことを原則としている。「管理者」という広い区分だけでは過剰閲覧になる。

**推奨対応**

- 通常の受付担当・運営担当には回答原文を表示しない
- 診断サポート専用権限を設けるか、本人からの明示的な問い合わせ時だけ閲覧可能にする
- 閲覧理由の入力、監査ログ、閲覧日時、閲覧者を記録する
- 一覧・分析画面では原文ではなく匿名化・集計値を使用する
- システム管理者であっても業務上不要な原文閲覧を許可しない

### C-06: AIへ送信するデータの同意・最小化・保存方針が不足している

診断4問と自由記述には、センシティブな個人情報が含まれる可能性がある。Phase 2の入力項目だけでは、外部AIへ送る前の安全条件が定義されていない。

**Phase 2開始前に必要な決定**

- AI処理に関する同意文と同意バージョン
- AIへ氏名、電話番号、LINE ID、参加者番号などを送らないこと
- 自由記述の個人情報検知・削減方法
- Provider側のデータ保持・学習利用条件
- プロンプト、入力、出力、エラーの保存期間
- ログに回答原文やAPIキーを残さないこと
- 削除請求・同意撤回時の扱い
- 再生成できる担当者と回数制限

### C-07: AI固定文フォールバックの品質基準が不足している

AI障害時の固定レポートは必須だが、回答内容と無関係な断定文を表示すると、参加者に誤解を与える可能性がある。

**推奨対応**

- 回答を評価・断定しない安全なテンプレートを用意する
- 8感情カテゴリおよび4軸ごとに決定的に構成する
- AI生成結果と固定文結果を画面上で過度に区別せず、品質差による不利益を避ける
- フォールバック結果にもテンプレートバージョンを記録する

### C-08: 8感情の固定範囲と表示ラベルの版管理が不足している

内部コードを固定し、表示ラベルをサービスごとに変更する方針は妥当である。ただし、ラベル変更が過去イベントの回答解釈を変えないようにする必要がある。

**推奨対応**

- 内部コード8種は共通の不変値とする
- 表示ラベルはテンプレートバージョンまたはイベントスナップショットへ保存する
- 複合感情は基本感情コードとの多対多マッピングとして版管理する
- 分析結果には使用したカードセット・ラベルセットの版を記録する

### C-09: Phase 6での共通化では遅い

新指示書ではSHIME® Engine共通化がPhase 6に置かれている。しかし、Phase 1〜5を婚活専用構造で実装してから共通化すると、大規模な再設計が必要になる。

**推奨対応**

- Phase 1からProvider interface、module key、versioned configuration、tenant scopeを採用する
- 婚活固有の文言と画面はサービスモジュール側に置く
- Phase 6は「共通化の開始」ではなく「他サービスから利用可能な状態への仕上げ」とする

### C-10: 24時間チャットの開始・終了条件が未定義

「24時間後」の起点が不明であり、結果確定取消、通知遅延、参加者の初回アクセス時刻との関係も決まっていない。

**確認が必要な内容**

- 起点を運営の結果公開時刻、個別通知時刻、初回入室時刻のどれにするか
- Asia/Tokyo表示とUTC保存
- 結果確定を取り消した場合の即時停止
- 24時間経過後の閲覧可否とデータ保存期間
- ブロック時に相手へ見せる表示
- 通報対応者、対応期限、証跡保存
- 退会・アカウント削除時のメッセージの扱い

### C-11: チャットの安全・利用規約が不足している

ブロック・通報機能だけでは運用できない。イベント後の参加者間通信を提供するため、規約、禁止事項、通報対応、保存期間が必要である。

**推奨対応**

- チャット専用利用規約と同意履歴を用意する
- 個人情報、勧誘、迷惑行為、脅迫等の禁止事項を明記する
- 通報受付後の管理画面、停止、証跡、対応記録を実装範囲に含める
- 管理者が通常時に私信を自由閲覧できない権限設計にする

### C-12: 1週間後フォローの送信条件が不足している

送信基準、対象者、重複防止、配信停止の扱いが未定義である。

**確認が必要な内容**

- イベント終了から7日後か、結果公開から7日後か
- 成立者・不成立者の両方へ送るか
- LINE連携解除、ブロック、通知停止時の扱い
- 再送条件とdedupe key
- 回答保存期間と管理者の閲覧範囲

### C-13: 90日プログラムの商品条件をハードコードする危険がある

商品名、49,800円、7日間無料などが仕様に記載されているが、共通OSではコードへ直接埋め込まない。

**推奨対応**

- 商品、価格、通貨、税込区分、無料期間、課金条件をバージョン付き設定として管理する
- 申込時点の商品条件スナップショットを保存する
- 後日の価格変更が既存申込へ影響しないようにする
- 決済Providerをinterfaceの背後に置く

### C-14: 決済事業者・法務・取消条件が未確定

Phase 5は決済Provider、特定商取引法表示、利用規約、プライバシーポリシー、返金・解約条件の確定なしには開始できない。

**Phase 5開始前に必要な決定**

- 決済事業者と利用アカウント
- 3Dセキュア等の本人認証方針
- 無料期間終了と初回課金の正確な日時計算
- 解約期限、解約の即時／期間末適用
- 返金条件
- Webhook署名検証、冪等性、再送処理
- 特定商取引法に基づく表示
- 商品提供主体と問い合わせ窓口

### C-15: イベント参加者と継続プログラム利用者の本人・同意境界が不足している

イベント参加を根拠に、継続プログラムへ自動加入させたり、イベント用同意を転用したりしてはならない。

**推奨対応**

- イベント参加とプログラム加入を別の契約・同意として管理する
- プログラム利用者IDとイベント参加者IDの関連付けを明示的に行う
- イベント削除・保存期限とプログラムデータ保持を分離する
- 未加入者にはプログラム機能の権限を付与しない

### C-16: 既存機能が「実装済み」であることと「本番運用可能」であることが混在している

指示書冒頭の既存機能一覧は、コード上の存在を示しているが、本番環境設定、実機LINE確認、負荷確認、復元訓練、運用リハーサルの完了を意味しない。

**推奨対応**

Concierge Phase 0では各項目を次の状態に分けて記録する。

- コード実装済み
- 自動テスト済み
- ステージング手動確認済み
- 本番設定済み
- 本番相当リハーサル済み

### C-17: 2026-08-08本番準備との優先順位が不明

新指示書のP0と、既存イベントOSの本番準備P0が同じ優先度名になっている。新機能開発によって、正式イベント情報、LINE本番設定、定期ジョブ、監視、バックアップ、総合リハーサルを遅らせる危険がある。

**確認が必要な内容**

- Concierge Phase 1〜3を2026-08-08で使用するか
- 使用しない場合は本番イベント後の別リリースとするか
- 使用する場合の機能フラグと即時無効化手順
- Event OS本番準備とConcierge開発の担当・期限・Go/No-Go判定を分けること

### C-18: テストコマンドに統合テストが明記されていない

新指示書にはlint、typecheck、test、build、E2Eがあるが、現行プロジェクトの必須チェックにはintegration testsも含まれる。

**各Phaseで実施するチェック**

```bash
pnpm lint
pnpm typecheck
pnpm test:unit
pnpm test:integration
pnpm build
pnpm test:e2e
```

リポジトリの`pnpm test`がunit・integrationの両方を含む場合でも、PR本文では実行対象を明示する。

### C-19: Migrationの「ロールバック方法」を安全に定義する必要がある

本番データを持つMigrationで単純なdown migrationを実施すると、診断回答、メモ、チャット等を失う可能性がある。

**推奨対応**

- 原則として後方互換Migrationと機能フラグによる停止を優先する
- カラム・テーブル削除は別Phaseで行う
- PRのロールバック欄にはアプリの切戻し、機能停止、forward-fix、バックアップ復元条件を記載する
- 本番データを破棄するrollbackを自動実行しない

### C-20: 商標、診断表現、医療・心理的表現の確認が必要

「SHIME®」「AIレポート」「恋愛タイプ」「婚活ウェルビーイング」等の表現は、商標、誤認、心理診断・医療行為との混同を避ける必要がある。

**推奨対応**

- 正式な商標表記ルールを確認する
- 診断ではなく自己理解支援であり、医学的・心理学的診断ではない旨を表示する
- 結婚、交際、マッチング結果を保証しない
- AI生成物の注意事項と問い合わせ先を規約・画面に記載する

## 5. データ設計でPhase 0中に確認する事項

### 5.1 既存データの調査

- `emotion_card_sets`、`emotion_cards`、`emotion_selections`の現在の用途と参照箇所
- `questionnaires`系テーブルが席案内5問専用として分離されていること
- Dream、Love Passport、希望入力、結果表示の公開範囲
- participantとuserの関係、およびイベントをまたぐアカウント利用可否
- 通知ジョブのdedupe、再送、失敗監視
- 既存権限で私的回答を閲覧できる経路の有無
- 監査ログへPIIや回答原文が保存されていないこと
- カード画像の保存Provider、公開URL、削除、版管理

### 5.2 新規テーブル案の評価条件

新規テーブル名はPhase 0で確定する。少なくとも次を満たす。

- 既存テーブルの名前・意味と衝突しない
- `tenant_id`、`event_id`、必要な`service_module`境界を持つ
- participant本人以外の参照を権限で制限する
- テンプレート、バージョン、イベントスナップショットを分離する
- AIジョブに冪等キーを持たせ、重複生成・重複課金を防止する
- メモは本人と相手の組み合わせ、トーク回を一意に識別できる
- チャットは確定済みmatch以外から参加できない
- 決済WebhookはProviderイベントIDで重複処理を防止する
- 保存期限と削除状態を追跡できる
- `created_at`、`updated_at`をUTCで保存する

## 6. API・画面でPhase 0中に確認する事項

### API

- 既存のLIFF参加者セッションを診断へ安全に利用できるか
- eventIdだけでなくtenantとparticipantを必ず検証しているか
- 診断途中保存と提出後ロックの状態遷移
- AIレポート生成の冪等性、タイムアウト、再試行、固定文切替
- メモ自動保存のupsert条件と競合処理
- 希望入力APIがメモから自動選択されないこと
- チャット送信時のmatch、member、期限、block、停止の検査順
- フォロー通知と課金Webhookのdedupe

### 画面

- Dreamと診断を明確に区別した参加者導線
- 320px、375px、390pxで横スクロールが発生しないこと
- 戻る・再読込・LIFF再起動で入力が失われないこと
- メモ自動保存の成功・失敗・再試行状態が分かること
- 希望入力時にメモを確認しても入力状態とスクロール位置が失われないこと
- チャットの終了時刻、通報、ブロック、退出を分かりやすく表示すること
