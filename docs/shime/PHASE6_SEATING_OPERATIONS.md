# Phase 6 席案内・配置運用

## 事前設定

- イベントに5問の質問票バージョンを割り当てる。
- 参加区分の組合せ、会話スタイル補完性、テーブル・席、接触回避／同行者／既知関係を管理データとして登録する。
- 順序尺度の選択肢コードは数値（例: `1`〜`5`）を使い、「相手に合わせたい」は `flexible` とする。

利用中の質問票バージョンは直接編集せず、新しいバージョンを作ってイベントへ切り替える。イベント開始後の切替は禁止する。

## 当日手順

1. 受付状況と欠席・接触回避情報を確定する。
2. 管理画面の「新しい配置案を生成」で下書きを作る。
3. 未配置警告と席数を確認し、必要な席をロック／入替／未配置化する。
4. 欠席変更後は新しい配置案を再生成する。
5. manager が確認ダイアログを経て公開する。

スマートフォンでは、参加者番号の前方一致または氏名の部分一致で対象を検索し、配置済み・未配置・ロック中で絞り込む。席変更は参加者カード内で行い、使用中の席は選択できない。変更件数を確認して保存した後にだけ公開できる。保存・公開中、通信失敗、ログイン期限切れは操作位置の近くへ表示する。

最新の配置案は「テーブル別の座席図」で確認する。配置済み、固定、空席を色と枠で区別し、席変更と固定操作は保存前でも座席図へ即時反映される。以前の下書きは初期状態で閉じ、最新案と取り違えないようにする。

公開確認にはイベント名、配置済み人数、未配置人数を表示する。未保存の変更がある間は公開操作を無効にする。

公開前の下書きは参加者APIへ返らない。公開後は `/api/liff/events/:eventId/seat` から本人のテーブル・席だけを返し、他参加者の回答やDreamは返さない。

## 計算とフォールバック

- アルゴリズムは `deterministic-v1`。同じスナップショットからは同じ結果を返す。
- 未回答／「答えたくない」の軸を除外し、残りの重みを100へ正規化する。
- AIは席決定に使用しない。案内理由は固定文を使用する。
- 障害時は公開済み席表をCSV／紙へ転記して運用する。
