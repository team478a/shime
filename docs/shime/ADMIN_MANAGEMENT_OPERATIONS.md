# 管理機能拡張 運用ガイド

更新日: 2026-07-14

## 管理者・権限

`system_admin` は管理トップの「管理者・権限」からスタッフを追加・更新できる。

- ログインID、表示名、初期パスワード、権限を指定して追加する。
- 権限は `reception`、`operator`、`manager`、`system_admin` の4種類。
- アカウント状態は有効、ロック、無効を選択できる。
- パスワード、権限または有効状態を変更すると、対象者の既存セッションを失効する。
- 自分自身の `system_admin` 権限を外す操作と、最後の有効な `system_admin` を無効化する操作は拒否する。
- パスワード平文は保存・監査ログ出力しない。

## テーブル・席マスター

`manager` または `system_admin` はイベントの「テーブル・席」からテーブルコード、定員、席コードを設定できる。

テーブル・席は会場共通ではなく、イベント単位で設定する。初期設定では次のいずれかで保存前の下書きを作成する。

1. 「かんたん自動生成」でテーブル記号、開始番号、テーブル数、1卓の席数を入力する。
2. CSVひな形を取得し、`table_code`、`capacity`、`seat_code`を1席1行で記入して読み込む。
3. テーブルごとの詳細編集で、不均等な席数や席コードを調整する。
4. 座席図プレビュー、テーブル数、総席数、イベント定員との差を確認する。
5. 確認ダイアログを経て保存する。自動生成とCSV読込だけではデータベースへ保存されない。

テナント全体を管理できる `manager` または `system_admin` は、確認中の構成を「会場レイアウトテンプレート」として保存できる。同じテンプレート名で保存すると旧版を保持したまま新しい版を作成し、選択肢には最新版だけを表示する。

初期設定では次のコピー元を選択できる。

- テナント内の最新版会場レイアウトテンプレート
- 同じテナントの過去イベント（直近20件のうちテーブル設定があるもの）

コピーは画面上の下書きを置き換えるだけで、保存確認が完了するまで対象イベントへ反映しない。コピー後の個別編集はコピー元へ逆反映しない。イベント限定権限のスタッフは、テナント全体へ影響するテンプレート作成と他イベント参照を行えない。

管理トップの「会場テンプレート」では、テナント全体を管理できる `manager` または `system_admin` が次を確認できる。

- テンプレートごとの最新版、全版履歴、卓数・席数と座席構成プレビュー
- 各版をイベントへ保存した回数、最後に適用したイベントと日時
- 利用中の最新版のアーカイブ（理由入力と確認必須）

テンプレートからイベントのテーブル・席を実際に保存した時点で、元テンプレートID・版、保存後の構成スナップショット、SHA-256、実行者、日時を適用履歴へ保存する。アーカイブは新規コピー元から外すだけで、過去版、適用履歴、コピー済みイベントを削除・変更しない。復元は既存版の上書きではなく、新しい版として作成する。

アーカイブ済みテンプレートの最新版には「この版から新しい版を作成」を表示する。この操作は名前の再入力を要求せず、同じテンプレートキーの次版として内容を複製する。イベントで有効なテンプレートをコピーした場合も、テンプレート保存欄へコピー元の正確な名前を自動設定し、似た名前の別系列が意図せず作られることを防ぐ。

- 席コードはイベント内で一意にする。
- テーブル定員より多い席は登録できない。
- 総席数がイベント定員を超える場合は警告する。
- 席配置案で使用済みの席マスターは置換できない。新しい配置案を作る前に確定する。
- 保存操作と件数は監査ログへ記録する。
- CSVひな形は `/templates/TABLE_IMPORT_TEMPLATE.csv` から取得する。
- 配置案が作成済みのイベントからテンプレート保存はできるが、そのイベント自身のテーブル・席マスターは置換できない。

## 申込フォーム項目

`manager` または `system_admin` はイベントの「申込項目」から表示名、入力種類、必須区分、表示順、選択肢を設定できる。

- 必須、任意、非表示を選択できる。
- 氏名、参加区分、規約同意、プライバシー同意などの保護項目は非表示にできない。
- 電話またはメールの少なくとも一方を表示する。
- 保存後は公開申込フォームAPIへ反映する。

## イベント削除

イベント削除は `system_admin` のみ実行できる。

- 対象が下書き状態であること。
- イベントコードの再入力と削除理由を必須とする。
- 参加申込、CSV取込、参加者、同意、通知、席配置または結果データがある場合は削除を拒否する。
- 削除したイベントのコード、名称、状態、理由をテナント監査ログへ残す。

本番運用データが存在するイベントは削除せず、状態管理と保存期間ポリシーで扱う。
